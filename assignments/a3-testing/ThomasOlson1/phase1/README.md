I went through each spec clause by clause, focusing on what each clause actually promises versus what it marks "outside the spec." That part mattered because asserting on disclaimed behavior — like LRU non-int capacity or float endpoints in interval_merger — would fail the clean run and tank my whole Phase 1 score. I used Claude as a tool to draft test code once I'd decided what to test, to brainstorm edge cases I'd missed, and to check what the clean module actually did before pinning a value. I didn't use it to read the spec for me or decide what tests to include — when it suggested tests pinning "outside the spec" behavior or redundant cases, I pushed back.

The 140 tests cover every clause across the three modules:

- **lru_cache (39):** capacity validation, put/get, TTL behavior in both directions, re-put TTL replacement with timer-reset, capacity eviction with expired-slot interaction, use-tracking with order preserved through expiration and eviction, ghost re-put, silent-expiry len, hashable keys, negative TTL.
- **interval_merger (34):** start>end validation, output sort, merge semantics at every boundary (overlap, touching, adjacent), chain merges, zero-length at every boundary, empty input, input-mutation snapshot test.
- **cart (67):** add_item validation, apply_code with case sensitivity, SAVE conflict preserving first, stacking, each code's behavior, BOGO qty boundaries, FREESHIP threshold trio, FLAT5-after-percent ordering, clamp at 0, multi-SKU end-to-end, banker's rounding alternation, empty cart per code.

The tests I'm most proud of are the ones that span multiple clauses at once. The best example is checking whether the order of applying save codes creates rounding issues: at a subtotal where SAVE10 hits the banker's-rounding boundary, applying SAVE20 afterward (which gets rejected by the conflict rule) shouldn't perturb SAVE10's effect. That test only works because it spans the stacking rule and the rounding rule at the same time. The interval_merger sort-then-swap test is similar in spirit — a reversed interval mixed into unsorted input still has to raise ValueError, not get silently normalized while the function sorts and merges around it.

The straightforward tests were easy — direct spec readings translate into assertions, and Claude could implement them cleanly once I described what I wanted. What was harder was thinking of all the edge cases on my own. But once I prompted Claude with my initial thoughts, it could usually expand from there and help me find more when I needed them.
